# 情境化 SOP 执行助手（移动端 + AI 监护 + 操作助手）— 设计方案 (v2)

> 日期：2026-06-12
> 状态：设计完成 v2（D1–D21 对齐；按"SOP 是上下文网中心"重估，新增情境聚合 + 操作助手 Copilot 两根支柱）
> 关联系统：SmartSOP（既有的纯 SOP 编写/管理系统，Vue3 + FastAPI + MySQL）、上游 smart CMMS

## 1. 背景与目标

SmartSOP 当前是一个**刻意解耦的纯 SOP 编写/管理系统**（程序 → 章节 → 步骤的树形结构），不含"执行"环节。

**核心洞察**：一个可执行的 SOP 不是孤立清单，它是一张**上下文网的中心**——关联着上下游 SOP、设备档案、历史经验反馈、相关仪表读数。因此本项目的定位不只是"执行+监护"，而是一个 **情境化 SOP 执行助手**：把 SOP 放回它的上下文网里，**辅助操作者顺利完成、避免人因失误、并在需要时给出"有据可依"的实时建议**。

### 四根支柱 + 一个闭环（D17）
| 支柱 | 性质 | 一句话 |
|---|---|---|
| ① 引导执行 + 电子记录 | 地基 | 拆步执行、逐步打钩、可追溯记录、签名 |
| ② AI 监护防错 | **推**（被动盯、主动警） | 眼镜+麦克风实时核查，软提醒纠偏 |
| ③ 情境聚合 | 连接组织 | 把上下游 SOP/设备档案/历史经验/仪表读数聚合成「执行上下文包」 |
| ④ 操作助手 Copilot | **拉**（操作者问、AI 答） | 基于上下文包的带护栏 RAG 建议，不脱离 SOP 上下文 |
| 闭环：经验反馈 | 学习 | 执行记录/偏差 → 沉淀+专家审核 → 知识库 → 下次执行的上下文 |

**统一 AI 原则**：监护（推）与助手（拉）共享同一份执行上下文，且**全是建议、有据可查、人最终决定**（D4 升级为全系统原则）。

### 目标
- 跑通"编写 → 执行 → 监护 → 记录"端到端闭环，并把执行嵌入其上下文网。
- 执行记录满足 **ISO（9001/IATF 等）质量体系**要求：受控、可检索、可追溯到人、留存可控（D8）。
- 所有 AI 输出（监护告警 + Copilot 建议）均为辅助、标注来源、人决策。

### 非目标（本期不做）
- 不改造 SmartSOP 的核心编写能力（仅做元数据扩展："监护期望" + "SOP 参考关系(编写/执行参考)" + "图纸/附件"）。
- 不做用户/角色/审批工作流的重建（沿用上游 CMMS 身份，按需对接）。
- AI 不做"硬拦截"式强制阻断（D4）。
- **不接 IoT/SCADA 实时数据源**：仪表读数靠人工读取/OCR（D20）。

## 2. 关键设计决策

| 编号 | 决策 | 选择 | 理由 |
|---|---|---|---|
| D1 | 监护核心机制 | **AI 实时识别防错** | 最高价值路线；以软提醒形式落地 |
| D2 | AI 核查类型 | 读数/文字 OCR、物体/工具/零件、动作/步骤完成性、安全/合规（四类全要，分期上） | 按成熟度排期：OCR + 安全合规先行 |
| D3 | 现场设备 | **双设备**：手机端=执行+打钩+签名；智能眼镜/摄像头+麦克风=监护（看画面 + 听语音） | 手机解放为控制面，眼镜做第一视角与语音 |
| D4 | AI 判定处置 | **软提醒为主，人最终决定** | AI 必有误判；高亮疑点 + 要求确认 + 疑点入档，不阻流 |
| D5 | 系统形态 | **独立服务**，只读消费 SmartSOP，保持其纯净 | 实时/GPU 负载与编写系统隔离，延续解耦哲学 |
| D6 | AI 计算部署 | **边云混合（C）**，第一期边缘做薄、重活先放云端 VLM | 兼顾能力/隐私/时延，渐进下沉，避免一期被边缘训练拖死 |
| D7 | 执行/监护关系 | **执行为主干，监护为可插拔叠加层**，两者独立会话 | 支撑"纯手机/监护"模式自由切换与优雅降级 |
| D8 | 合规等级 | **一般工业 / ISO 体系**（非制药 GMP / Part 11） | 记录受控+可追溯即可；电子签名走轻量化，不做 Part 11 加密签名 |
| D9 | 身份来源 | **对接上游 CMMS / 企业 SSO**，本服务不自建账号 | 企业身份背书使轻量签名可信；角色尽量从上游取 |
| D10 | 手机端形态 | **Flutter**（跨平台原生，单一 Dart 代码库） | 双平台 NFC/BLE/离线/推送全支持，化解 PWA 的 iOS 限制 |
| D11 | 时延模型 | **尽量实时，不接受事后补判**；每步当场必有处置 | 用户要求弱网也实时 → 实时核查下沉边缘，难判 VLM 尽力实时、弱网当场转人工 |
| D12 | 边缘算力落点 | **配对手机当边缘节点**（Flutter + 端侧 TFLite） | 眼镜瘦化为采集+串流；实时核查不出本地链路最稳；复用 Flutter 端侧 ML |
| D13 | 任务发起 | **以 CMMS 工单驱动为主**，绑定工单号、完成回写 | 最符合 CMMS 语境，追溯最清晰；归档视图复用 SmartSOP ReportLab |
| D14 | 脱敏策略 | **默认只模糊人脸**，手机端脱敏，原始流不离设备 | 对监护友好、上下文最全、隐私中等；按需录而非全程录 |
| D15 | 协作模型 | **单活动操作者 + 中途交接**，每步记录实际操作人 | 最简会话/签名模型；不做双人见证/并行/一监护多任务（远程协助留第二期单路） |
| D16 | 语义纠偏 | 新核查模式**「实时操作对象防错」**：端侧手部-对象交互比对，**更醒目软提醒(不破 D4)+偏差事件入档**；框架覆盖错对象/跳步/禁止动作，**先做错对象** | 高后果偏差(如开错阀门)需实时→必须端侧；系统拦不住人手，故最醒目告警但不硬拦 |
| D17 | 产品定位 | **情境化 SOP 执行助手**：四支柱(执行/监护/情境聚合/Copilot)+经验闭环；全系统 AI 原则=有据/建议/人决定 | SOP 是上下文网中心，监护(推)+助手(拉)共享同一上下文 |
| D18 | 执行上下文包 | 工单触发任务时**聚合并预取**：本 SOP+上下游 SOP+设备档案+历史经验+相关读数，下发设备**可离线读** | 一举三得：操作者随手查、喂监护判更准、做 Copilot 事实地基；是 D7"执行上下文"的长大 |
| D19 | 操作助手 Copilot | **带护栏 RAG**：仅从上下文包检索作答、**每条标注来源**、超范围明说"联系班组长"、建议不自动执行；走**云端拉取**，与端侧实时监护不冲突 | "不脱离 SOP 上下文"是安全红线；拉取式可容忍几秒延迟 |
| D20 | 仪表读数源 | **人工读取/OCR**，不接 IoT/SCADA；上下文里的读数取自历次执行 OCR 抓取 + 手工 | 监护产生数据→喂回上下文→喂 Copilot，自成闭环零外部集成 |
| D21 | 历史经验来源 | **自动沉淀 + 专家审核**：从执行记录/偏差自动提候选，专家审核补充入库 | bootstrap：早期薄、越跑越厚；human-in-the-loop 保质量 |
| D22 | SOP 上下文边界 | SOP 新增属性：**编写参考SOP / 执行参考SOP / 图纸+附件文档**；它们 + 设备档案 + 本SOP经验库 + 相关CR = **Copilot 可引用的明确边界**（边界外拒答） | "不脱离 SOP 上下文"靠 DB 关系圈定而非提示词；图纸是多模态上下文；编写参考=背景，执行参考/图纸=主源 |
| D23 | 经验反馈架构 | **两者并用、统一知识层**：B 内联经验库(每 SOP 独立、直接喂 Copilot) + A **Condition Report 模块**(事件入口、带 triage→结案闭环、引用 SOP/步骤/设备)；知识层**两种范围(SOP / 设备)** | B 做知识面、A 做事件入口；偏差事件可一键升级 CR，CR 结案经审核蒸馏入经验库 → 闭环合上 |
| D24 | CR 流程 | **轻量持续改进捕获**(非正式 CAPA)；**班组长/主管处置**；生命周期 待审阅→处理中→已结案(采纳/关闭)；**结案双出口**：采纳→蒸馏经验、SOP缺陷→**一键生成 SOP 修订请求回 SmartSOP 编写端** | 轻流一线乐用；CR→SOP 改版正式联动，现场发现真正反哺程序 |
| D25 | 内联经验录入 | 步骤级+SOP级、**语音优先**录入、轻结构(类型+正文+附证)、可一键升级为 CR；**两级可见性：未审核对人可见(标"未核实")，Copilot 只引用已审核** | 现场友好捕获，又不让未核实提示变成 AI 建议 |
| D26 | Copilot 交互 | **上下文锚定**(默认知当前步)；**语音为主 + 预设问题兑底 + 文字**；答案简短带引用、可 TTS；超边界拒答给"联系班组长/转人工/发起CR"出口；**离线降级为本地检索原文**(在线=生成,离线=检索)；问答留痕 | 现场噪声/手套权衡；预设问题由 AI 从步骤上下文自动生成+人工补 |

### D10 手机端细化
- **端分工**：Vue3 既有 Web = 编写端（监护期望标注面板放此）；Flutter = 移动执行 App（现场操作者用）。两端不共享代码。
- **能力**：`nfc_manager`(工牌再确认，双平台)、`flutter_blue_plus`(BLE)、本地 SQLite/Hive(健壮离线)、FCM/APNs(推送)、`camera`/扫码。
- **眼镜通信**：眼镜低延迟**本地串流给手机**（手机即边缘节点，见 D12），不再走"眼镜→云"中转。

### D11/D12 时延与边缘细化
- **分级（cascade）控成本**：高频确定性核查(安全/PPE、物体状态、读数 OCR)放**手机端侧 ML**，100ms~1s 实时出、弱网也出；云 VLM **仅离散触发**处理难判语义(双核对/动作判定)，绝不按帧喂。
- **"不接受事后补判"= 每步当场必有处置**：云 VLM 网络够→几秒内给；网络不行→**当场立即转人工/操作者现场确认**（实时响应，非延后）。没有"待会儿再判"。
- **眼镜→手机串流**：需眼镜支持低延迟本地串流(WiFi-Direct/WiFi-Aware/USB-C 量级，BLE 带宽不够)；手机需具备端侧推理算力(NPU)。这是 #4 眼镜选型的硬约束。
- **成本量级**：VLM 离散触发，目标每任务 VLM 调用数控制在个位~几十次；端侧推理无按次费用。

### D8 合规细化（ISO 体系）
- **可放宽**：电子签名 = 「已认证身份 + 服务端时间戳 + 确认动作」，不做 Part 11 的密码学绑定/签名含义声明；"绝对不可篡改"放宽为**"记录受控 + 修改留痕"**（默认只读，确需修正时不抹除原值、记录谁改/改什么/为何改，满足 ISO §7.5 记录控制）。
- **仍保留**：可追溯到人、可信时间戳、对应工单/设备/SOP 版本、记录可检索、留存期限可定义、媒体证据哈希防误改。

### D9 身份对接细化
- 新服务作为 relying party，接 CMMS/企业 **SSO（OIDC/SAML）或 CMMS API** 获取身份与角色。
- **现场多人共用设备**场景：签名/打钩前做一次低成本身份再确认（**PIN 或工牌 NFC 轻点**），保证"可追溯到本人"。

### 4.1 实施载体（Surfaces）

本系统横跨**三个载体**，文档描述整体，按功能分工落地：

| 载体 | 角色 | 承载的设计 |
|---|---|---|
| **Web 端**（既有 Vue3 SmartSOP，桌面） | 编写/管理/审核 | SOP 编写(既有)；监护期望标注(D3)；语义纠偏"正确操作对象"标注(D16)；SOP 参考关系+图纸/附件(D22)；经验候选专家审核(D21/D23)；Condition Report triage/结案；执行记录单查看/检索/导出+监护覆盖率报表(§10)；后台配置(留存/脱敏/风险等级) |
| **移动端**（Flutter，现场） | 执行/监护/查阅/问答 | 步序引导/打钩/电子签名+PIN/工牌(①、D9)；端侧 ML 实时监护(D12)；软提醒/偏差告警响应(D4/D16)；上下文包离线查阅(D18)；Copilot 问答(D19)；随手记内联经验+发起/升级 CR(D23)；手机端脱敏(D14)；眼镜配对/串流(D12) |
| **云端后端**（独立服务，无界面） | 引擎 | 执行编排引擎、监护协调器、云 VLM、情境聚合层、Copilot RAG、知识库、记录与审计、媒体存储(D5；组件表标"云服务"者) |

一句话：**移动端=现场执行面，Web 端=编写管理面（复用 Vue3），后端=所有业务逻辑/AI/记录/知识库（独立服务）。** 三端对应 D10(Flutter)、D10细化(Vue3 编写端)、D5(独立服务)。

## 5. 解耦与执行模式

**核心原则：执行是主干，监护是可插拔的叠加层；两者是各自独立的会话，靠共享的"执行上下文"绑定。**

执行编排引擎完全不依赖监护。手机端打钩有无 AI 结果都能独立完成。由此天然支持三种模式：

| 模式 | 设备 | 行为 | 适用 |
|---|---|---|---|
| 纯手机模式 | 仅手机 | 手动逐步执行 + 电子记录，无 AI，**离线可跑** | 低风险、无眼镜、网络差 |
| 监护模式 | 手机 + 眼镜 | 手机执行 + 眼镜 AI 软提醒叠加 | 关键/高风险任务 |
| （可选）远程协助 | 手机 + 眼镜 + 远程人 | 在监护模式上叠加远程人工 | 疑难/争议步骤 |

保证真解耦的三点：
1. **监护会话可中途挂载/卸载**：执行中可随时激活/关闭眼镜，监护会话 attach/detach 到当前执行上下文。
2. **优雅降级**：眼镜没电/掉线时自动退回纯手机模式，执行不中断，记录标注"监护中断"。
3. **监护期望是可选增强，不是依赖**：无眼镜时该元数据被忽略或降级为"拍照自证"，SOP 不因无眼镜而不可执行。

随之而来：执行记录必须如实记录**每一步当时的模式**，以支撑"监护覆盖率"追溯。

## 6. 数据模型（关键实体）

**编写侧（SmartSOP 扩展）**

组织方式：**一个步骤挂 0..N 个"核查点"**（一步可能既要读数又要查 PPE），每个核查点一条监护期望。

公共字段（所有核查类型）：

| 字段 | 说明 |
|---|---|
| step_id | 关联步骤 |
| 核查类型 | OCR / 物体 / 动作 / 安全 / **语义纠偏(防错)** |
| 模态 | 视觉 / 语音 / 双模（看+听，利用麦克风） |
| 严重度 | 提示 / 警告 / **紧急偏差**（均为软提醒，决定告警醒目度；紧急偏差=最醒目多模态告警，仍不硬拦） |
| 触发时机 | 进入该步时 / 操作者主动触发 / 该步全程持续 |
| 操作者提示语 | 如"请将压力表对准镜头" |
| 证据留存 | 是否抓拍/录制留证入档 |
| 置信度阈值 | 低于则判"看不清"→提示补拍（默认值，可空） |

类型专属字段：

| 类型 | 专属字段 | 期望值来源 |
|---|---|---|
| 读数/OCR | 目标描述、匹配方式(精确/包含/正则/数值范围)、单位 | 固定值 / 数值范围 / 引用工单变量 / 仅记录不判定 |
| 物体/工具/状态 | 目标物(可关联物料编码)、判定(期望存在/不存在/状态=X) | 如"阀门=关闭"、"指示灯=绿" |
| 动作/完成性 | 期望动作(受控动作词表)、完成判据(持续N秒/覆盖区域/次数) | 第一期降级为"录制片段+操作者自证" |
| 安全/合规 | 期望项(PPE清单/上锁挂牌/人员不在危险区)、检查时点 | 持续 or 进入步骤时 |
| **语义纠偏(防错)** | 子型(错对象/跳步/禁止动作)、**正确操作对象**、易混对象(可选)、禁止项(禁止动作用)、偏差级别 | 详见 §8；**第一期只做"错对象"**，框架预留另两类 |

**语音双核对**（模态=双模时）：要求操作者口述关键信息，系统做语音转写 + 画面 OCR 双重核对，两路一致才通过——把"念出来"变成一道免费交叉校验，降低读错/抄错。

**编写端标注易用性**：步骤编辑器旁"监护"面板 `+ 添加核查点`；表单按类型动态展开；期望值可下拉引用工单变量；提供预设模板一键套用；（进阶）可挑历史证据图做 few-shot 参考样例。

**执行+监护服务（新库，与 SmartSOP 分离）**

| 实体 | 关键字段 | 要点 |
|---|---|---|
| 执行任务 | procedure_id + **版本快照**、操作者、状态、模式、起止时间、监护覆盖率 | 执行时快照当时 SOP 版本，改版不影响历史记录 |
| 步骤执行记录 | task_id、step_id、**当时模式**、操作者、打钩时间、结果(完成/跳过/NA)、人工确认、备注 | 一步一条；"当时模式"支撑覆盖率追溯 |
| AI 核查结果 | step 记录关联、核查类型、判定(通过/疑点/看不清)、置信度、证据指针、模型版本、时延 | 软提醒数据来源，全部入档 |
| 偏差事件 | task_id、step_id、偏差子型(错对象/跳步/禁止动作)、期望对象、实际对象、时间、告警已读、操作者处置、证据 | 语义纠偏命中即记一条；进归档"异常清单"，供事后追责/改进 |
| 监护会话 | task_id、眼镜设备、attach/detach 时间、覆盖步骤范围、中断事件 | 记录眼镜挂载/卸载/降级 |
| 媒体证据 | task_id、step_id、类型(视频/音频/抓拍)、存储URI、哈希、脱敏标记 | 对象存储；哈希保证不可篡改 |
| 审计与签名 | 已认证身份、服务端时间戳、确认动作、修改留痕(who/what/when/why) | ISO 级轻量签名；受控修改而非绝对不可改 |
| SOP 参考关系 | source_sop、target_sop、**关系类型(编写参考/执行参考/上游/下游/相关)**、说明 | SmartSOP 侧扩展；执行参考=操作主源，编写参考=背景源(D22) |
| 图纸/附件文档 | 关联SOP(或步骤)、类型(图纸/手册/表单)、存储URI、是否多模态可读 | SOP 属性扩展；图纸供 VLM 多模态引用(D22) |
| 经验条目 | **范围(SOP/设备)**、关联设备/SOP/步骤、类型(提示/坑点/窍门/澄清/警告)、内容、附证、来源(内联/自动沉淀/专家/CR蒸馏)、**状态(未审核/已审核)**、来源链接 | 统一知识层；**Copilot 只引用已审核**(D25)；SOP范围喂Copilot、设备范围解孤岛(D23) |
| Condition Report (CR) | 编号、来源(现场/偏差升级/复盘)、发起人、时间、类型(异常/未遂/SOP缺陷/设备问题/改进)、严重度、**引用(SOP/步骤/设备/工单)**、描述、证据、状态(待审阅→处理中→已结案[采纳/关闭])、处置人、结案出口(蒸馏经验/转SOP修订/关闭理由) | 轻量持续改进(D24)；班组长处置；偏差事件可一键升级；结案蒸馏入经验库 |
| SOP 修订请求 | 关联SOP、来源CR、问题描述、状态(待办→处理→完成) | SmartSOP 侧；CR 结案"SOP缺陷"一键生成，带 CR 引用入编写端待办(D24) |
| Copilot 问答记录 | task_id、step_id、提问、回答、**引用来源(边界内)**、时间、操作者 | 可审计(问了什么/答了什么/据何而答)，也是改进与经验沉淀的素材 |

## 7. 单步执行 + 监护时序

```
眼镜            手机(边缘节点, Flutter)            监护协调器/云VLM
 │采集(看+听)
 │──低延迟串流──►│ 显示步骤N+监护期望
 │              │ 端侧ML实时检测/OCR
 │              ├─[安全/状态/读数]当场出结果──→即时软提醒(不依赖网络)
 │              │ 需难判语义? ──WebSocket──────►│ 调度云VLM核查
 │              │                ◄──结果(尽力实时)─┤
 │              │ [网络不行]→当场立即转人工/操作者现场确认(不延后)
 │              ├─[通过]→可打钩 [疑点]→高亮要确认 [看不清]→提示补拍
 │              │─操作者打钩/语音确认─►编排引擎记录(步骤·时间·人·模式·AI结果·确认·证据)→受控记录
 │              └─全部完成→生成执行记录→电子签名→归档
```

纯手机模式下，监护各步跳过，手机端打钩 → 记录链路照常闭环。

## 9. 技术选型倾向（待拍板）

- **手机端 + 边缘节点**：**Flutter + 端侧 ML（TFLite/MediaPipe）**（见 D10/D12）。
- **端侧模型**：检测器(安全/PPE/物体状态) + 轻量 OCR，目标实时、弱网可用。
- **云 AI**：多模态大模型（VLM）做难判语义核查，离散触发、尽力实时。
- **实时通道**：眼镜↔手机本地串流(WiFi-Direct/Aware) + 手机↔云 WebSocket。
- **新服务后端**：倾向沿用 FastAPI 技术栈以复用团队经验（待定）。

### 时延/成本目标（SLO）
- 端侧实时类(安全/状态/OCR) < 1s，**不依赖网络**；
- 云 VLM 难判类目标几秒内，超时/弱网当场转人工，**不延后**；
- 成本纪律：VLM 离散触发，每任务个位~几十次量级。

## 10. 执行记录产物与 CMMS 对接

产物分三层：**规范记录（真相源）→ 归档视图（给人看/审计）→ 下游回写（喂回 CMMS）**。

**A. 规范记录**：数据模型里的结构化记录（任务/步骤/AI结果/确认/签名/证据指针/模式/覆盖率），**库为准**，永久可查可检索。

**B. 归档视图「执行记录单」**：按需把规范记录渲染成只读 PDF/HTML，**复用 SmartSOP 既有 ReportLab 能力**：
- 抬头：工单号、SOP 名+版本、操作者、监护人、起止时间、执行模式、监护覆盖率、最终签名；
- 正文：逐步表格（步骤/结果/时间/人/AI判定/人工确认/证据缩略图）；
- 异常清单（审计最关心）：AI 疑点、跳过步骤、监护中断段、转人工记录；
- 媒体：关键抓拍缩略图内嵌；完整视频/音频不内嵌，按 URI + 哈希引用。

**C. CMMS 对接（D13：以工单驱动为主）**：
- 入口：CMMS 工单 → 触发执行任务、绑定工单号；
- 出口：完成 → 回写工单（状态 完成/异常 + 记录单链接 + 关键指标：耗时/异常数/监护覆盖率），走 REST/webhook。

**D. 留存与检索**：留存期可配置（记录与媒体可分别设）；按 工单/设备/人/SOP/时间 检索、可导出。

## 11. 隐私脱敏与协作模型

**隐私脱敏（D14）**
- 脱敏在**手机端（边缘节点）**完成，原始未脱敏流不离设备，上云/入库均为脱敏后；
- 默认**只模糊人脸**（非操作者优先）；
- **按需录**（核查窗口/关键步骤），不全程录像；
- 语音：双核对**转写后默认丢弃原始音频**，仅留转写文本（可配）；
- 眼镜**录制指示灯**告知旁人；
- 媒体访问**受控 + 审计**，仅授权角色可看；留存期可配（媒体通常短于记录）。

**协作模型（D15）**
- 一个任务**同时只有一个活动操作者**；
- 可**中途交接（接班）**，交接为显式动作，每步记录实际操作人；
- 不做双人见证复签 / 同任务并行 / 一监护人盯多任务（远程协助第二期再做单路版）；
- 会话与签名模型保持简单。

## 12. 语义纠偏（实时操作对象防错）

**定位**：已有监护期望是**正向验证**（"该做的做对了吗"）；语义纠偏是**反向纠偏**（"是不是正在做错的/不该做的事"）。作为一种新核查模式接入同一架构，不是新子系统。覆盖三类偏差：错对象、跳步/错顺序、禁止动作（**第一期只做错对象**，框架预留另两类）。

**核心机制（以"开错阀门"为例）**——要"紧急/实时"，必须端侧出（云端来不及）：
1. 编写端：步骤声明"正确操作对象 = 阀门 V1"（+可选易混对象 V2/V3）；
2. 端侧实时(手机 TFLite)：手部追踪(MediaPipe Hands) + 目标物检测 + 标签 OCR → 识别"操作者此刻正在接触/操作的对象"；
3. 比对：实际对象 V2 ≠ 期望 V1 → 命中；
4. 命中 → 立即本地**最醒目软提醒**（震动+声音+红屏 / 眼镜端"停！这是 V2，应操作 V1"）+ 记一条**偏差事件**入档；
5. 云端 VLM 仅在端侧"拿不准这是哪个阀"时辅助识别，不在告警主链路。

**告警强度（D16，遵从 D4）**：紧急偏差 = 最醒目多模态软提醒，但**不强制确认、不拦打钩、不物理硬拦**（系统拦不住人手）；价值在"当场喊停 + 留痕追责"。

**误报控制**：仅在"实质接触/操作"（抓握/转动）时触发，不在"路过/瞟一眼"时响；置信度不足时降级为轻提示"确认你在操作 V1？"。

**泛化（框架预留）**：跳步/错顺序——系统已知当前步，检测到越步动作即告警（成本低）；禁止动作——编写端给步骤加"禁止项"字段。

**分期**：放 **第二期**（依赖物体识别 + 手部追踪，建立在第二期对象识别之上）；如需提前到第一期需评估端侧手部追踪工程量。

## 13. 情境聚合与操作助手 Copilot（支柱③④）

### 13.1 执行上下文包（D18）
工单触发任务时，系统聚合并**预取**一个上下文包到设备（可离线读）：

```
执行上下文包（随任务下发，可离线读）= Copilot 的引用边界（D22）
├─ 本 SOP（版本快照）              ← 已有
├─ 执行参考 SOP（操作中要用：上游隔离/子程序/下游）← SOP 参考关系  ┐主源
├─ 图纸 + 附件文档（P&ID/接线图/手册，多模态可读）← SOP 附件      ┘
├─ 设备档案（型号/履历/已知问题）   ← 读 CMMS                      ┐主源
├─ 本SOP经验库（老师傅提示/坑点）   ← 知识库(SOP范围经验条目)        │
├─ 相关 Condition Report           ← CR 模块(引用本SOP/设备)        ┘
├─ 编写参考 SOP（为什么这么写）     ← SOP 参考关系                  …背景源
└─ 相关仪表读数（历史/手工）        ← 历次执行 OCR 沉淀 + 手工（D20，不接 IoT）
```
一举三得：① 操作者随手查上下文（离线也行）；② 喂监护让 AI 判更准；③ 做 Copilot 的事实地基。

### 13.2 操作助手 Copilot（D19/D22）—— "不脱离 SOP 上下文"的护栏
维护现场一句幻觉指令可能出事，故 Copilot = **带护栏的 RAG**：
- **检索范围严格限定在上下文包（=D22 的边界）**，靠 DB 关系圈定、不靠提示词；编写参考=背景源，执行参考/图纸/设备档案/经验库=主源；**经验库只引用「已审核」条目**（D25 两级可见性，未核实提示不入 Copilot 语料）；
- **每条建议标注来源**（引用哪个 SOP/步骤/图纸/设备履历/读数/经验条目/CR）；
- **超边界明说**："SOP 未涵盖，建议联系班组长"，不外推；
- **建议永不自动执行**，操作者判断（同 D4）；
- 走**云端拉取**：对话式、容忍几秒延迟，**与端侧实时监护互不冲突**。

**交互（D26）**
- **上下文锚定**：默认知道操作者在第几步，问题自动带当前步/设备/期望范围，不用每次交代背景；
- **语音为主 + 预设问题兑底 + 文字**：噪声大时退回"按步骤常见问题一键问"；**预设问题由 AI 从该步上下文自动生成 + 可人工补**（不加编写负担）；
- **答案**：简短优先可展开、每条带可点开的来源引用、可 TTS 免提听、涉安全显著标注且重申"建议非指令"；
- **超边界拒答**：明说"SOP 未涵盖"，并给**一键联系班组长 / 转人工 / 发起 CR** 出口；
- **离线降级**：在线=生成式问答；弱网=**本地检索原文**（直接给相关 SOP 段落/已审核经验条目，不做生成总结），不至失能；
- **建议不自动执行**（D4），可"存证本次问答"；问答全程留痕（可审计、喂经验沉淀）。

### 13.3 经验反馈：两者并用、统一知识层（D23）
**统一知识层 = 一个底座、两种范围**：`SOP 范围经验库`（每 SOP 独立，直接喂 Copilot）+ `设备范围知识`（解"这台泵老漏"这类跨 SOP 的孤岛）。两个范围上下文包都拉。

两条入口分工：
- **B 内联经验（知识面）**：操作者在步骤/SOP 位置随手记经验（提示/坑点）→ 进该 SOP 经验库，轻量、即用；
- **A Condition Report（事件入口）**：异常/未遂/SOP缺陷/设备问题/改进走 CR，**引用 SOP/步骤/设备**，带 triage→处理→结案闭环；**监护偏差事件可一键升级为 CR**。

闭环：执行记录/偏差/CR/操作者备注/Copilot 问答 → **自动提经验候选** →（CR 结案蒸馏）→ **专家审核**（D21）→ 经验条目入库（按 SOP/设备范围）→ 进入后续任务上下文包。**bootstrap：早期薄，越跑越厚**；读数也由监护 OCR 持续沉淀，数据反哺上下文，自成闭环。

### 13.4 Condition Report 模块（D24，轻量持续改进捕获）

**生命周期（3 态 + 结案二分）**
```
待审阅 ──班组长 triage──► 处理中 ──► 已结案
 (提交)                            ├─ 已采纳 → 蒸馏经验 / (SOP缺陷)转 SOP 修订请求
                                  └─ 已关闭(不采纳) + 理由
       旁路动作：可「转交」工艺/设备员（跨班组/工艺问题），非独立状态
```
非正式 CAPA、无强制 RCA、无审批 —— 符合"轻"。

**字段**：自动带入(编号/发起人/时间/来源/引用 SOP·步骤·设备·工单/模式) + 发起人填(标题/类型/严重度/描述/证据) + 处置人填(结论/是否采纳/蒸馏经验[范围 SOP/设备]/是否转 SOP 修订/关闭理由)。

**发起入口**：移动端随手发起 / 监护偏差事件一键升级(D23) / 执行记录复盘；Web 端可补录。

**结案双出口**：① 采纳 → 蒸馏经验条目(范围 SOP/设备) → 专家审核(D21)入知识库 → 喂 Copilot；② SOP 缺陷 → **一键生成 SOP 修订请求，带 CR 引用回 SmartSOP 编写端待办**(D24)；③ 不采纳 → 关闭 + 理由，留档可查。

**载体**：移动端 发起/升级/查看自己的 CR；Web 端 班组长 triage/处置/结案/蒸馏(§4.1)。

**追溯**：CR 唯一编号；引用链贯穿 SOP/步骤/设备/工单/偏差事件/媒体证据。

### 13.5 内联经验录入（D25，B 路径）

- **粒度**：步骤级 + SOP 级（步骤级最贴上下文，SOP 级放通用提示）。
- **时机/入口**：执行中随手（当前步）/ 执行后复盘 / 平时浏览 SOP 时。
- **方式（现场友好）**：**语音输入为主**（手套/手忙，可用眼镜麦克风免提记）+ 文字 + 拍照/录像附证（复用媒体证据）。
- **内容（轻结构）**：自动带当前步骤/设备/SOP；选类型（提示/坑点/窍门/澄清/警告）+ 正文 +（可选）附证。
- **与 CR 分流**：内联经验 = 知识/窍门、无需处理；若其实是要处理的问题，系统提示**"要不要升级成 CR？"**。
- **两级可见性（D25，安全红线）**：录入即对人可见但标 **"一线提示·未核实"**；**Copilot 只引用「已审核」条目**；专家（Web 端）审核后升级为已审核并入 Copilot 语料，可合并/精炼去重。
- **载体**：移动端录入；Web 端审核/精炼；执行时经验随上下文包呈现在该步旁（D18）。
